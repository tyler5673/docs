This page explains how to migrate your SDK code to the **new pattern** LangSmith supports for runs ingest.
It covers both the **Python** and **JavaScript/TypeScript** SDKs.
For API endpoint URLs and request/response changes, see [API v1 and v2 overview](/langsmith/api-v1-v2-overview).
For version and sunset policy, see [API versioning and support policy](/langsmith/endpoint-deprecation).

## Recommended migration process

1. **Upgrade** to a LangSmith SDK version that supports the new pattern: Python **≥0.5.0**, JavaScript/TypeScript **≥0.4.0**.
2. **Add `trace_id` and `dotted_order`** to every `Client.create_run` / `Client.createRun` and `Client.update_run` / `Client.updateRun` call (see [Quick reference](#quick-reference-new-pattern-vs-legacy) and code examples below).
3. **Keep batching enabled** default in both SDKs.
4. **Prefer high-level APIs when possible:** In JS, use `RunTree` or `traceable` so the SDK sets `trace_id`, `dotted_order`, and UUID7 for you.
5. **If you use Python’s `AsyncClient`:** It does not support the new pattern. Switch to the sync `Client` with batching or call `Client.batch_ingest_runs` / `Client.multipart_ingest` from your async code.

## SDK version requirements

| SDK | Minimum version for new pattern |
| --- | ------------------------------- |
| **Python** | 0.5.0 |
| **JavaScript/TypeScript** | 0.4.0 |

Install or upgrade to at least these versions. For deprecation and sunset dates for the legacy path, see [API versioning and support policy](/langsmith/endpoint-deprecation).

## Quick reference: new pattern vs legacy

Two things must both be true for the new pattern:

- **Payload:** `trace_id` and `dotted_order` are present (in the run for create, in the update for update).
- **Client:** Batching is enabled (`auto_batch_tracing=True` in Python, `autoBatchTracing` true in JS; both default to true).

If either is missing, the SDK uses the legacy path.

## Key changes

- **`Client.create_run` / `Client.createRun`:** With `trace_id`, `dotted_order`, and batching enabled → new pattern. Without either → legacy path.
- **`Client.update_run` / `Client.updateRun`:** Same rule — pass `trace_id` and `dotted_order` in the update and keep batching enabled for the new pattern.
- **`Client.batch_ingest_runs` / `Client.batchIngestRuns`:** Always new pattern. No migration needed.
- **`Client.multipart_ingest` / `Client.multipartIngestRuns`:** Always new pattern. No migration needed.
- **Python `AsyncClient.create_run` / `AsyncClient.update_run`:** Always legacy path. To use the new pattern, use the sync `Client` or call `Client.batch_ingest_runs` / `Client.multipart_ingest`.
- **JS:** No separate async client; the main `Client` is async and supports both patterns depending on payload and `autoBatchTracing`.

## Python SDK

For the new pattern, both must be true: `trace_id` and `dotted_order` in the payload, and `auto_batch_tracing=True` (default).

### Before and after: create_run and update_run

**Legacy (missing trace_id / dotted_order):**

```python
from langsmith import Client
from uuid import uuid4
import datetime

client = Client()
run_id = uuid4()
start_time = datetime.datetime.now(datetime.timezone.utc)

client.create_run(
    id=run_id,
    name="my_run",
    run_type="llm",
    inputs={"prompt": "hello"},
    start_time=start_time,
    project_name="my_project",
)

client.update_run(run_id, outputs={"result": "hi"})
```

**New pattern — Adding trace_id and dotted_order**

```python
from langsmith import Client, uuid7
import datetime

client = Client()
run_id = uuid7()
start_time = datetime.datetime.now(datetime.timezone.utc)
# Dotted order: start_time str + run_id (required for new pattern)
dotted_order = start_time.strftime("%Y%m%dT%H%M%S%fZ") + str(run_id)

client.create_run(
    id=run_id,
    trace_id=run_id,  # for root run, trace_id = run id
    dotted_order=dotted_order,
    name="my_run",
    run_type="llm",
    inputs={"prompt": "hello"},
    start_time=start_time,
    project_name="my_project",
)
client.update_run(
    run_id,
    outputs={"result": "hi"},
    trace_id=run_id,
    dotted_order=dotted_order,
)
```

### Methods that always use the new pattern

**`Client.batch_ingest_runs`** and **`Client.multipart_ingest`** — Always new pattern. Accept sequences of run creates and updates and ingest them in a single batched call. No migration needed.

### RunTree and traceable

**`RunTree`** and **`@traceable`** auto-generate **`trace_id`** and **`dotted_order`**, so with the default **`auto_batch_tracing=True`** on the sync **`Client`**, they always use the new pattern.

### AsyncClient

**`AsyncClient.create_run`** and **`AsyncClient.update_run`** — Always legacy pattern. No batching, no multipart, no new pattern. To use the new pattern from async code, switch to the sync **`Client`** or call **`Client.batch_ingest_runs`** / **`Client.multipart_ingest`** directly.

## JavaScript/TypeScript SDK

For the new pattern, all three must be true: `autoBatchTracing` is true (default), `trace_id` is present, and `dotted_order` is present.

### Before and after: createRun and updateRun

**Legacy (missing trace_id / dotted_order):**

```typescript
const client = new Client();
await client.createRun({
  name: "my-run",
  run_type: "chain",
  inputs: { question: "hello" },
  id: "some-uuid-v4",
});

await client.updateRun(runId, {
  outputs: { answer: "world" },
  end_time: Date.now(),
});
```

**New pattern — Adding trace_id and dotted_order**

```typescript
import { Client } from "langsmith";
import { uuid7 } from "langsmith/uuid";
import { convertToDottedOrderFormat } from "langsmith/run_trees";

const client = new Client();
const runId = uuid7();
const startTime = Date.now();
const { dottedOrder } = convertToDottedOrderFormat(startTime, runId);

await client.createRun({
  name: "my-run",
  run_type: "chain",
  inputs: { question: "hello" },
  id: runId,
  trace_id: runId,
  dotted_order: dottedOrder,
});

await client.updateRun(runId, {
  outputs: { answer: "world" },
  end_time: Date.now(),
  trace_id: runId,
  dotted_order: dottedOrder,
});
```

### Methods that always use the new pattern

**`Client.batchIngestRuns()`** — Always use the new pattern. Accepts arrays of run creates and run updates and ingests them in a single batched call. No migration needed.

**`Client.multipartIngestRuns()`** — Always use the new pattern. Same as `batchIngestRuns` but also supports attachments and gzip compression. Requires `trace_id` and `dotted_order` on every run (throws if missing). No migration needed.

### RunTree and traceable

**`RunTree.postRun()`** and **`RunTree.patchRun()`** — Always pass `trace_id` and `dotted_order` to the client, so with default `autoBatchTracing=true` they use the new pattern. **`traceable(fn)`** — Uses RunTree internally, so same behavior.

## Migration checklist

**Python**

- Use LangSmith Python SDK ≥0.5.0.
- For `Client.create_run`: pass `trace_id`, `dotted_order`, and `start_time`; keep `auto_batch_tracing=True`.
- For `Client.update_run`: pass `trace_id` and `dotted_order`.
- For `AsyncClient`: switch to sync `Client` or `batch_ingest_runs` / `multipart_ingest` if you need the new pattern.

**JavaScript/TypeScript**

- Use LangSmith JS SDK ≥0.4.0.
- For `Client.createRun()` / `Client.updateRun()`: include `trace_id` and `dotted_order`; use UUID7 for `id`; do not set `autoBatchTracing: false`.
- Prefer `RunTree` or `traceable` so the SDK sets `trace_id`, `dotted_order`, and UUID7 for you.
